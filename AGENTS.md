# AGENTS.md

Rules for AI agents implementing changes in TaskFlow. Read this before writing code.

## SDD workflow

1. Read constitution files + feature spec (`guide/user-story-task-dependencies.md`)
2. `/opsx:propose <change-name>` → review proposal, specs, design, tasks
3. `/opsx:apply` → implement `tasks.md` in order
4. Run tests before marking complete
5. `/opsx:archive` when done

Use CodeGraph (`codegraph explore`, MCP tools) to trace callers and impact before editing.

## Architecture (summary)

```
Backend:  routes.py → Controller → Manager → Model
Frontend: page → hook → lib/api.ts → FastAPI
```

Never skip layers. Never put business logic in routes or controllers.

## Migrations — critical

**Do not create migration files manually.**

1. Edit SQLAlchemy models in `backend/app/models/`
2. Export new models in `models/__init__.py`
3. Ensure models are imported in `backend/alembic/env.py`
4. Generate:

```bash
make dev-migrate-create MSG="describe your change"
```

5. Review the autogenerated file in `backend/alembic/versions/`
6. Apply:

```bash
make dev-migrate
```

**Windows:** see [guide/local-development.md](guide/local-development.md) for `conda run` equivalents.

Seed data goes in a **separate** Alembic migration, not in Python scripts.

## Development commands

| Action | macOS / Linux | Windows (PowerShell) |
| ------ | ------------- | -------------------- |
| Start app | `make dev` | `.\scripts\dev-local.ps1` |
| New migration | `make dev-migrate-create MSG="..."` | `cd backend; conda run -n task-flow alembic revision --autogenerate -m "..."` |
| Apply migrations | `make dev-migrate` | `cd backend; conda run -n task-flow alembic upgrade head` |
| Run tests | `make dev-test` | See [guide/local-development.md](guide/local-development.md) |
| Reset DB | `make dev-reset` | `docker compose down -v; docker compose up db -d` + migrate |

## Code quality checklist

Before finishing any task:

- [ ] Follows layer architecture (no logic in wrong layer)
- [ ] Migration autogenerated and applied (if schema changed)
- [ ] Google-style docstrings on new public Python functions
- [ ] `lib/api.ts` updated for new endpoints
- [ ] Types mirror backend schemas
- [ ] Backend tests for happy path + error cases
- [ ] Frontend tests for UI requirements
- [ ] `make dev-test` passes
- [ ] No hardcoded API URLs
- [ ] No `/api` prefix on routes

## API convention

Root-level paths: `/projects`, `/tasks`, `/projects/{id}/tasks`. Status codes: `201` create, `204` delete, `404` not found, `422` business rule violation.

## Styling

Light theme, cobalt + slate, DM Sans. Use semantic Tailwind tokens (`page`, `accent`, `status-*`). See `frontend-standards.md`.

## Do not

- Hand-write Alembic migrations for schema changes
- Add business logic to routes or controllers
- Use SQLite or raw SQL without design justification
- Add React Query, Redux, or new state libraries
- Push to protected branches (`main`, `workshop/*`) — commit locally, owner pushes
- Commit `.codegraph/` (local index, gitignored)
