# Migrations

PostgreSQL schema changes use **Alembic autogenerate**. Migrations live in `backend/alembic/versions/`.

Workshop runtime is **full Docker only** — all Alembic commands run inside containers.

## Golden rules

1. **Add or change SQLAlchemy models first** — then generate the migration.
2. **Never hand-write a migration file from scratch** for schema changes.
3. **Never edit the database manually** — always go through Alembic.
4. **Import new models in `backend/alembic/env.py`** so autogenerate detects them.

## Workflow

### macOS / Linux

```bash
# 1. Edit models in backend/app/models/
# 2. Generate migration (stack must be runnable)
make migrate-create MSG="describe your change"

# 3. Review the generated file in backend/alembic/versions/
# 4. Apply
make migrate

# 5. Rebuild so containers pick up model + migration code
make up
```

### Windows (PowerShell)

```powershell
docker compose run --rm `
  -v "${PWD}/backend/app:/app/app" `
  -v "${PWD}/backend/alembic:/app/alembic" `
  backend alembic revision --autogenerate -m "describe your change"
docker compose exec backend alembic upgrade head
docker compose up --build -d
```

Full command table: [guide/docker.md](../../guide/docker.md).

## After generating

- **Review** the autogenerated `upgrade()` / `downgrade()` — Alembic can miss renames or enum changes.
- **Seed data** belongs in a separate migration (see `e478ddcffa15_seed_demo_data.py`), not in application code.
- **Rebuild** with `make up` after model/migration changes (containers do not hot-reload).

## Reset (development only)

```bash
make reset    # wipe DB volumes + rebuild stack
```

Wipes the DB volume and re-applies all migrations including seed.

## In OpenSpec tasks

Every schema change task must include:

1. Model change
2. `make migrate-create MSG="..."`
3. Review migration file
4. `make migrate`
5. `make up` and verify API starts without errors
